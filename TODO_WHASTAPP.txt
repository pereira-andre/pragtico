# Plano WhatsApp WOW

Objetivo: levar o canal WhatsApp de `resposta inbound em texto` para `assistente operacional proativo`, com briefings automáticos, alertas úteis e ações rápidas seguras no próprio chat.

## Leitura do estado atual

O repo já tem a base certa:

- adaptador Cloud API em `integrations/whatsapp_cloud.py`
- webhook e persistência do canal em `blueprints/whatsapp.py`
- runtime comum em `core/chat_runtime.py`
- feedback por reação `👍/👎`
- bloqueio explícito de mutações no WhatsApp

Isto confirma que a arquitetura v1 está feita. O próximo salto não deve ser "mais texto". Deve ser "mais produto".

## Aposta principal

Se tivermos de escolher só uma direção para criar efeito `WOW`, a aposta deve ser:

1. `briefings proativos`
2. `alertas operacionais automáticos`
3. `quick actions` seguras no WhatsApp

Em vez de esperar que o utilizador pergunte, o PRAGtico passa a chegar ao utilizador com contexto pronto:

- resumo de turno às `06:00`
- alterações relevantes em ETA/ETD
- aviso local novo com impacto operacional
- janela de maré / meteorologia / ondulação resumida
- botões/listas para abrir ações úteis sem obrigar a escrever comandos longos

## Resultado esperado do v1 WOW

Primeira versão a entregar:

- briefing de turno enviado automaticamente para números subscritos
- 1 ou 2 tipos de alerta realmente úteis
- mensagem com formato mais forte que texto corrido
- entrada no chat via `quick replies` ou `list message`
- sem ações destrutivas diretas no portal

Exemplo:

```text
PRAGtico | Briefing 06:00

Hoje:
- 3 entradas
- 2 saídas
- 1 manobra crítica
- maré útil até 09:40
- ondulação moderada

Alertas:
- NAVIO X atrasou +45 min
- Aviso local novo no Canal Norte

[Ver escalas críticas]
[Marés]
[Meteorologia]
```

## Princípios de implementação

- manter a lógica de negócio principal fora do adaptador WhatsApp
- não abrir mutações completas logo na fase 1
- usar proatividade só para eventos com valor operacional claro
- tornar cada envio observável: quem recebeu, quê, quando, estado Meta
- começar com baixo risco e expandir por camadas

## Fase 1: base de mensagens ricas

Objetivo: deixar de enviar apenas `text` e passar a suportar mensagens com estrutura operacional.

### Entregas

- suporte a `template message` para outbound iniciado pelo sistema
- suporte a `interactive message`
- suporte a `reply buttons` e `list messages`
- normalização das respostas interativas recebidas

### Ficheiros a mexer

- `integrations/whatsapp_cloud.py`
- `blueprints/whatsapp.py`
- `tests/test_operational_flow.py`

### Mudanças técnicas

- extrair um método genérico de envio, por exemplo `send_message(payload)`
- manter `send_text_message(...)` como wrapper simples
- adicionar:
  - `send_template_message(...)`
  - `send_interactive_buttons(...)`
  - `send_interactive_list(...)`
- estender `parse_webhook_events(...)` para reconhecer:
  - `interactive.button_reply`
  - `interactive.list_reply`
- converter respostas interativas em eventos internos consistentes

### Critério de aceitação

- o sistema consegue enviar uma mensagem com botões
- o clique num botão entra no webhook e fica normalizado
- o fluxo fica testado de ponta a ponta

## Fase 2: serviço interno de outbound

Objetivo: separar o webhook inbound do envio proativo.

### Entregas

- endpoint interno para envio de mensagens WhatsApp
- payload interno consistente para texto, template e interativo
- registo uniforme de eventos outbound

### Ficheiros a mexer

- `blueprints/whatsapp.py`
- `app.py`
- possivelmente novo módulo `core/whatsapp_outbound.py`
- `tests/test_operational_flow.py`
- `tests/test_storage.py`

### API interna recomendada

```text
POST /internal/channels/whatsapp/send
```

Payload mínimo:

```json
{
  "to_number": "3519...",
  "message_kind": "text|template|interactive_buttons|interactive_list",
  "template_name": "daily_briefing_pt",
  "body": {},
  "metadata": {
    "campaign": "shift_briefing",
    "trigger": "scheduled"
  }
}
```

### Critério de aceitação

- qualquer parte do sistema consegue pedir um envio outbound sem duplicar lógica Meta
- o outbound fica registado com `external_message_id`, estado e metadata útil

## Fase 3: preferências e subscrições

Objetivo: controlar quem recebe o quê.

### Entregas

- preferências por número WhatsApp
- subscrição por tipo de briefing/alerta
- janela horária e papel operacional

### Implementação recomendada

Começar simples. Em vez de criar já uma tabela complexa, guardar preferências em storage como estado de canal por utilizador/número:

- `whatsapp_enabled`
- `briefing_opt_in`
- `alert_eta_opt_in`
- `alert_warning_opt_in`
- `shift_label`
- `quiet_hours`

Se o modelo atual de storage não encaixar bem, criar uma estrutura explícita de `channel subscriptions`.

### Ficheiros a mexer

- `storage/local.py`
- `storage/postgres.py`
- `storage/base.py`
- `tests/test_storage.py`

### Critério de aceitação

- o sistema só envia briefings/alertas a números subscritos
- a subscrição fica persistida e consultável

## Fase 4: briefing diário

Objetivo: entregar a primeira funcionalidade que já parece produto acabado.

### Conteúdo do briefing

- chegadas previstas no período
- saídas previstas no período
- manobras com maior criticidade
- janela útil de maré
- resumo meteorológico
- resumo de ondulação
- avisos locais novos ou ativos relevantes

### Implementação recomendada

Criar um construtor dedicado, por exemplo:

- `domain/whatsapp_briefings.py`

Funções possíveis:

- `build_shift_briefing(now, role, scope) -> dict`
- `render_shift_briefing_text(...) -> str`
- `build_shift_briefing_actions(...) -> list`

### Scheduling

Há duas hipóteses válidas:

1. usar scheduler interno semelhante ao refresh de dados vivos
2. usar job externo do Railway a chamar o endpoint interno

Recomendação:

- v1: job externo do Railway para maior previsibilidade operacional
- v2: só trazer para scheduler interno se houver motivo forte

### Ficheiros a mexer

- novo `domain/whatsapp_briefings.py`
- `app.py`
- possivelmente `core/whatsapp_outbound.py`
- `tests/test_operational_flow.py`

### Critério de aceitação

- um briefing pode ser gerado deterministicamente
- o briefing pode ser disparado manualmente e por agendamento
- o utilizador recebe uma mensagem útil e curta, não um bloco caótico

## Fase 5: motor de alertas operacionais

Objetivo: tornar o canal realmente indispensável.

### Alertas a priorizar

Prioridade alta:

1. atraso relevante de ETA/ETD
2. novo aviso local com impacto
3. mudança de janela operacional / restrição importante

Prioridade média:

1. mudança de cais
2. alteração relevante de rebocadores
3. risco meteo/ondulação acima de limiar

Prioridade baixa:

1. resumos informativos sem urgência

### Regra de produto

Cada alerta precisa de:

- motivo objetivo
- alvo certo
- deduplicação
- cooldown
- ação seguinte clara

### Implementação recomendada

Criar um módulo próprio, por exemplo:

- `domain/whatsapp_alerts.py`

Capacidades:

- detetar diffs relevantes
- gerar fingerprint do alerta
- bloquear duplicados por janela temporal
- montar payload curto para WhatsApp

### Estado mínimo a guardar

- `alert_type`
- `target_number`
- `fingerprint`
- `triggered_at`
- `delivery_status`

### Critério de aceitação

- o mesmo alerta não dispara repetidamente sem motivo
- cada alerta tem texto curto e ação recomendada

## Fase 6: quick actions seguras

Objetivo: reduzir fricção e aproximar o canal da operação real.

### Escopo recomendado

Abrir primeiro só ações seguras:

- `Ver escalas críticas`
- `Marés agora`
- `Meteorologia`
- `Ondulação`
- `Avisos locais`
- `Resumo do navio X`

Só depois abrir ações operacionais com confirmação forte:

- `Validar manobra`
- `Preparar aprovação`
- `Preparar registo`

Não abrir já:

- criação/apagamento direto sem confirmação reforçada
- mutações amplas para qualquer número

### Implementação recomendada

As quick actions devem entrar como intents normalizadas no webhook e cair no runtime com contexto explícito.

Opções:

1. mapear botão/lista para pseudo-comando interno
2. mapear para texto normalizado tipo `/mares hoje`

Recomendação:

- v1 usar texto normalizado simples
- v2 só depois criar intents dedicadas se houver necessidade

### Ficheiros a mexer

- `blueprints/whatsapp.py`
- `core/chat_runtime.py`
- possivelmente `domain/chat_actions.py`
- `tests/test_operational_flow.py`

### Critério de aceitação

- uma quick action produz resposta correta sem o utilizador escrever nada
- o histórico continua coerente no storage

## Fase 7: observabilidade e controlo

Objetivo: tornar o canal operável em produção.

### Entregas

- métricas básicas por tipo de envio
- taxa de entrega/leitura/falha
- lista de últimos briefings/alertas enviados
- motivo do trigger
- reenvio manual de campanhas

### Implementação recomendada

Começar por aproveitar o que já existe em `channel_events` e enriquecer metadata.

Se ficar curto, criar vista simples na área admin:

- estado do canal WhatsApp
- últimos envios
- últimos falhanços
- subscrições ativas

### Ficheiros a mexer

- `blueprints/admin.py`
- templates admin relevantes
- `storage/*`
- testes de integração

### Critério de aceitação

- em produção é possível perceber rapidamente o que foi enviado e o que falhou

## MVP recomendado

Se o objetivo for impacto rápido com risco controlado, o MVP deve ser este:

1. suporte a `interactive buttons`
2. endpoint interno de outbound
3. briefing diário de turno
4. alertas de:
   - atraso ETA/ETD acima de limiar
   - novo aviso local relevante
5. quick actions:
   - `Marés`
   - `Meteorologia`
   - `Escalas críticas`

Isto já muda a perceção do produto de forma clara.

## Ordem de implementação sugerida

### Sprint 1

- mensagens `interactive`
- parser de replies
- endpoint interno outbound
- testes de envio e receção

### Sprint 2

- modelo de subscrições
- gerador de briefing diário
- disparo manual e agendado

### Sprint 3

- motor de alertas ETA/ETD
- alertas de avisos locais
- deduplicação e cooldown

### Sprint 4

- quick actions seguras
- observabilidade admin
- ajustes de copy e formato

## Variáveis de ambiente novas

```env
WHATSAPP_OUTBOUND_ENABLED=1
WHATSAPP_BRIEFING_ENABLED=1
WHATSAPP_BRIEFING_TIME=06:00
WHATSAPP_ALERTS_ENABLED=1
WHATSAPP_ALERT_ETA_MINUTES=30
WHATSAPP_ALERT_COOLDOWN_MINUTES=120
WHATSAPP_INTERNAL_SEND_TOKEN=segredo-interno
```

Se forem usados templates Meta:

```env
WHATSAPP_TEMPLATE_BRIEFING_PT=daily_briefing_pt
WHATSAPP_TEMPLATE_ALERT_PT=ops_alert_pt
```

## Riscos principais

- ruído operacional se os alertas forem excessivos
- outbound proativo sem subscrições claras
- quick actions que pareçam executar mais do que executam
- dependência de templates Meta antes de provar valor do fluxo

## Decisões de produto importantes

- `WOW` não vem de abrir tudo; vem de abrir a coisa certa
- primeiro valor deve ser `proatividade útil`, não `comandos complexos`
- o canal deve parecer um copiloto operacional, não um clone pobre do chat web

## Recomendação final

O corte certo para começar é:

1. `briefing diário`
2. `alerta ETA/ETD`
3. `alerta aviso local`
4. `3 quick actions` úteis

Se isto ficar bom, o WhatsApp passa de "temos integração" para "isto já faz parte da operação".
